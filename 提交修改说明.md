## 一、前置准备

### 1.1 首次使用 Git,配置用户信息(全局,只做一次)

```powershell
git config --global user.name "你的名字"
git config --global user.email "你的邮箱@example.com"
```

### 1.2 推荐使用 SSH(避免每次输密码)

```powershell
# 1. 生成 SSH 密钥(一路回车即可)
ssh-keygen -t rsa -b 4096

# 2. 复制公钥内容(手动添加到 GitHub Settings -> SSH and GPG keys -> New SSH key)
cat ~/.ssh/id_rsa.pub

# 3. 把远程地址从 HTTPS 切换为 SSH
git remote set-url origin git@github.com:asce-12138/KK-s_Wonder_Adventure.git
```

### 1.3 进入项目目录

```powershell
cd e:\kk_wonder_adventure\KK-s_Wonder_Adventure
```

## 二、标准提交流程(4 步)

### 第 1 步:查看当前修改状态

```powershell
git status
```

输出示例:

```
On branch main
Changes not staged for commit:
  modified:   src/main.py          # 已修改
Untracked files:
  src/modules/new_feature.py       # 新增未跟踪
```

### 第 2 步:把改动加入暂存区

**方式 A — 提交指定文件**(推荐,更安全)

```powershell
git add src/main.py
```

**方式 B — 提交所有改动**

```powershell
git add .
```

> ⚠️ `git add .` 会把当前目录下所有改动加入暂存区。提交前请确认 `.gitignore` 已正确忽略 `.venv/`、`__pycache__/`、`saves/` 等本地文件。

### 第 3 步:提交(在本地生成一个版本快照)

```powershell
git commit -m "feat: 新增玩家血条显示"
```

提交后可以用 `git log --oneline -3` 查看最近 3 条记录。

### 第 4 步:推送到 GitHub

```powershell
git push origin main
```

推送成功后,刷新 GitHub 仓库页面即可看到你的提交。

## 三、一行流快捷方式

熟练后可使用一条命令完成「暂存 + 提交 + 推送」:

```powershell
git add . && git commit -m "你的提交说明" && git push origin main
```

> 任何一步失败都会中断后续命令,便于排错。

## 四、常见问题与处理

### 4.1 推送被拒绝:远程有更新

```
! [rejected] main -> main (fetch first)
error: failed to push some refs to '...'
```

**原因**:你本地的版本落后于远程,需要先拉取。

**解决**:

```powershell
# 1. 先拉取远程最新代码
git pull origin main

# 2. 如果没有冲突,直接推送
git push origin main
```

### 4.2 拉取时出现冲突

**冲突标记示例**(`<<<<<<<`、`=======`、`>>>>>>>`):

```python
<<<<<<< HEAD
player_speed = 5
=======
player_speed = 8
>>>>>>> origin/main
```

**解决步骤**:

1. 打开冲突文件,手动选择保留哪段代码,删除所有冲突标记。
2. 解决完所有冲突后:
   ```powershell
   git add .
   git commit -m "merge: 解决与 main 的冲突"
   git push origin main
   ```

### 4.3 第一次推送这个分支

```powershell
git push -u origin main
```

`-u` 的作用:把本地 `main` 与远程 `main` 关联,以后只需 `git push` 即可。

### 4.4 误提交了不该上传的文件(例如 `.venv`)

如果文件**还没 push**:
```powershell
git rm --cached <文件名>
git commit -m "chore: 从版本控制中移除 <文件名>"
```

如果文件**已经 push**:
- 把文件名加入 `.gitignore`
- 之后从仓库中彻底删除(见 4.5)

### 4.5 从历史中彻底删除敏感文件 / 大文件

需要使用 `git filter-repo` 工具(慎用,会重写历史):

```powershell
pip install git-filter-repo
git filter-repo --path <文件路径> --invert-paths
git push origin main --force
```

> ⚠️ `--force` 会覆盖远程历史,操作前请与团队成员沟通。

### 4.6 撤销最后一次 commit(还没 push)

```powershell
# 保留修改内容,只撤销 commit
git reset --soft HEAD~1

# 彻底撤销 commit 和修改
git reset --hard HEAD~1
```

### 4.7 误把 `.venv` 一起 add 了

```powershell
# 先把 .venv 从暂存区移除(但保留在本地)
git rm --cached -r .venv
git restore --staged .venv

# 确认 .gitignore 中有 .venv/(项目已自带)
# 重新提交
git add .gitignore
git commit -m "chore: 忽略 .venv 目录"
```

## 五、Commit 消息规范

建议使用 **Conventional Commits** 规范,便于生成 changelog 和浏览历史。

### 5.1 格式

```
<类型>: <简短描述>

[可选的详细说明]

[可选的关联 Issue 编号]
```

### 5.2 常用类型

| 类型       | 含义                       | 示例                              |
| ---------- | -------------------------- | --------------------------------- |
| `feat`     | 新功能                     | `feat: 新增存档系统`              |
| `fix`      | 修复 bug                   | `fix: 修复玩家穿墙问题`           |
| `refactor` | 重构(既不是新功能也不是修 bug) | `refactor: 重构 player 模块` |
| `docs`     | 文档变更                   | `docs: 更新 README 启动说明`      |
| `style`    | 代码格式(空格、分号等)    | `style: 统一缩进为 4 空格`        |
| `test`     | 测试相关                   | `test: 添加 player 移动测试`      |
| `chore`    | 杂项(构建、依赖等)       | `chore: 升级 pygame 到 2.6.0`     |
| `perf`     | 性能优化                   | `perf: 优化碰撞检测算法`          |

### 5.3 优秀 vs 不好的 commit message

❌ 不推荐:
```
git commit -m "update"
git commit -m "fix bug"
git commit -m "改了一下"
```

✅ 推荐:
```
git commit -m "feat: 新增武器切换快捷键 (1/2/3)"
git commit -m "fix: 修复存档读取后玩家位置重置的问题"
git commit -m "refactor: 把 player.py 拆分为 movement 和 combat 两个模块"
```

## 六、常用命令速查表

| 用途                         | 命令                                  |
| ---------------------------- | ------------------------------------- |
| 查看当前状态                 | `git status`                          |
| 查看差异                     | `git diff`                            |
| 查看已暂存的差异             | `git diff --cached`                   |
| 暂存单个文件                 | `git add <文件>`                      |
| 暂存所有改动                 | `git add .`                           |
| 提交                         | `git commit -m "说明"`                |
| 推送                         | `git push origin main`                |
| 拉取并自动合并               | `git pull origin main`                |
| 查看提交历史                 | `git log --oneline`                   |
| 查看图形化分支历史           | `git log --oneline --graph --all`     |
| 撤销工作区修改               | `git checkout -- <文件>`              |
| 撤销暂存                     | `git restore --staged <文件>`         |
| 切换分支                     | `git switch <分支名>`                 |
| 创建并切换分支               | `git switch -c <新分支名>`            |
| 查看所有分支                 | `git branch -a`                       |
| 查看远程仓库地址             | `git remote -v`                       |

## 七、提交前自检清单 ✅

每次推送前过一遍:

- [ ] 代码在本地能正常运行(`python src/main.py` 启动游戏)
- [ ] `git status` 中没有意外文件(尤其是 `.venv/`、`__pycache__/`)
- [ ] commit message 描述清晰、符合规范
- [ ] 已先 `git pull` 同步过远程最新代码
- [ ] 冲突已全部解决
- [ ] 推送后到 GitHub 页面确认改动已显示
